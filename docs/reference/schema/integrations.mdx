# `integrations`

A typed-`Record<string, unknown>` bag. Tangly preserves whatever you put here on the parsed config and exposes it to:

- Custom layout components (via `Astro.locals` after manifest expansion).
- User scripts running in `theme/` overrides.
- The `tangly check` audit (which flags unknown top-level keys but accepts anything under `integrations`).

```json
{
  "integrations": {
    "intercom":   { "appId": "abc123" },
    "frontchat":  { "chatId": "xyz789" },
    "statuspage": { "url": "https://status.example.com" }
  }
}
```

## When to use this

- Per-deploy keys that aren't analytics (and so don't fit [`analytics`](/reference/schema/analytics)).
- Helpdesk / chat widget configs.
- Reading config from custom Astro components without hardcoding values.

## When NOT to use this

If Tangly already has a first-class field, prefer it. `analytics.posthog` is better than `integrations.posthog` — Tangly emits the script tag for you, with the right shape.
