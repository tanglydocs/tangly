# `tangly check`

Validates everything Tangly will need at build time — schema, nav references, internal links, image src, frontmatter, OpenAPI specs.

## Usage

```bash
tangly check [options]
```

## Options

<ParamField path="--strict" type="boolean">
  Warnings become errors. Use this in CI.
</ParamField>

<ParamField path="--no-links" type="boolean">
  Skip internal link checking (faster).
</ParamField>

<ParamField path="--json" type="boolean">
  Machine-readable output for CI integration.
</ParamField>

<ParamField path="--include-drafts" type="boolean">
  Include draft pages in checks.
</ParamField>

## What gets validated

<Steps>
  <Step title="docs.json schema">
    Parses against the published Zod schema. Errors include the JSON path and a suggestion when possible.
  </Step>
  <Step title="Nav references">
    Every page slug in `navigation` must resolve to an MDX file at `<root>/<slug>.mdx`.
  </Step>
  <Step title="Internal links">
    Every `[text](/path)` in MDX body resolves to a known page.
  </Step>
  <Step title="Image src">
    Every `<img src="...">` and Markdown image references a file on disk.
  </Step>
  <Step title="Frontmatter">
    Per-page frontmatter parses against the frontmatter Zod schema.
  </Step>
  <Step title="OpenAPI">
    Specs referenced via `api.openapi` parse via `@apidevtools/swagger-parser`.
  </Step>
</Steps>

## Exit codes

- `0` — all checks passed (or all issues are warnings without `--strict`).
- `1` — errors found.

## Example

```bash
$ bun x tangly check --strict
Pages: 47    Orphans: 2
  ⚠ docs.json: Nav references "guides/pricing" but no MDX file found
  ✗ guides/intro.mdx: Frontmatter missing required field "title"
```

Use `--json` to feed the output to a CI annotation script.
