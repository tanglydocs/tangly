# ResponseField

`<ResponseField>` documents one field of an API response. Its props mirror [`<ParamField>`](/reference/components/param-field) — the difference is presentational (no inbound location chip).

## Props

| Prop | Type | Default | Description |
|---|---|---|---|
| `name` | `string` | — | Field name. Use this when none of `path`/`query`/`body`/`header` apply. |
| `path` | `string` | — | Alternative name source. |
| `query` | `string` | — | Alternative name source. |
| `body` | `string` | — | Alternative name source. |
| `header` | `string` | — | Alternative name source (response headers). |
| `type` | `string` | — | Display-only type. |
| `required` | `boolean` | `false` | Marks as always-present. |
| `default` | `string` | — | Display-only default. |
| `deprecated` | `boolean` | `false` | Adds the deprecated chip. |

## Examples

### Top-level response field

```mdx
<ResponseField name="id" type="string" required>
  The created resource's UUID.
</ResponseField>
```

<ResponseField name="id" type="string" required>
  The created resource's UUID.
</ResponseField>

### Response header

```mdx
<ResponseField header="X-Request-Id" type="string">
  Request correlation ID. Echo back to support if you file an issue.
</ResponseField>
```

## Nested fields

`<ResponseField>` doesn't have built-in nesting. For nested object fields, use [`<Expandable>`](/reference/components/expandable) inside the description.

```mdx
<ResponseField name="user" type="User">
  The owning user.

  <Expandable title="User fields">
    <ResponseField name="user.id" type="string" />
    <ResponseField name="user.email" type="string" />
  </Expandable>
</ResponseField>
```
