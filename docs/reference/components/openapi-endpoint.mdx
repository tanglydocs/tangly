# OpenApiEndpoint

`<OpenApiEndpoint>` is Tangly's built-in OpenAPI viewer. It pulls a single operation out of a spec and renders it with the same layout you'd hand-roll using [`<ParamField>`](/reference/components/param-field) and [`<ResponseField>`](/reference/components/response-field) — but generated from the spec.

For most projects you don't author this tag yourself: setting `api.openapi` in `docs.json` and `openapi:` in page frontmatter auto-generates pages that wire up the right `<OpenApiEndpoint>` for each path/method. See [Authoring → OpenAPI](/guides/authoring/openapi).

## When to author manually

- The spec mixes endpoints across pages and you want one operation per section.
- You're augmenting a hand-written guide with a single live operation.
- The auto-generated layout doesn't match what you want.

## Props

| Prop | Type | Default | Description |
|---|---|---|---|
| `spec` | `string` | — | Path or URL to the OpenAPI spec. Required. |
| `method` | `string` | — | HTTP method. Case-insensitive. Required. |
| `path` | `string` | — | Operation path as it appears in the spec (`/v1/users/{id}`). Required. |
| `baseUrl` | `string` | (from spec) | Override the spec's `servers[0].url`. Falls back to `api.baseUrl` from `docs.json`. |
| `authMethod` | `"bearer" \| "basic" \| "key" \| "none"` | (from `api.auth`) | Authentication method for the try-it-out form. |
| `authName` | `string` | (from `api.auth`) | Header name when `authMethod="key"`. |

## Example

```mdx
<OpenApiEndpoint
  spec="/openapi.json"
  method="GET"
  path="/v1/users/{id}"
/>
```

## Picking a different viewer

If you'd rather use ReDoc, Scalar, or Stoplight, set `api.viewer` in `docs.json`:

```json
{
  "api": { "viewer": "scalar" }
}
```

Auto-generated pages then use [`<OpenApiScalar>`](/reference/components/openapi-scalar) instead of `<OpenApiEndpoint>`.
